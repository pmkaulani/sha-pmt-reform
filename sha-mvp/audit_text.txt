PART A — CRITICAL BUGS (Fix Before Any Presentation)



Bug A1 — Tharaka-Nithi County Name Mismatch (Affects Real People)

CRITICAL

Tharaka-Nithi residents get no ASAL semi-arid discount due to a spelling difference

AlgorithmSimulation.js line 25 defines 'Tharaka-Nithi' (hyphenated). App.jsx COUNTIES array uses "Tharaka Nithi" (space, no hyphen). When a user selects the county, d.county = "Tharaka Nithi" and ASAL_COUNTIES["Tharaka Nithi"] returns undefined. The 50% land/livestock discount never applies. A farmer with 2 acres gets KSh 160,000 land income instead of KSh 80,000 — potentially misclassified as non-indigent.

FIX: In App.jsx COUNTIES array, change "Tharaka Nithi" to "Tharaka-Nithi". Alternatively, normalize both to the same format. Also check Tharaka Nithi vs Tharaka-Nithi in all other string comparisons across the codebase.



// App.jsx COUNTIES — WRONG:

"Tharaka Nithi",   // produces ASAL_COUNTIES["Tharaka Nithi"] = undefined

// App.jsx COUNTIES — CORRECT:

"Tharaka-Nithi",   // matches ASAL_COUNTIES key, discount applies



Bug A2 — fraudRiskPercentile is Average Confidence, Not Cumulative Risk

CRITICAL

Multiple fraud flags do not increase the risk percentile — the formula averages, not compounds

AlgorithmSimulation.js line 940: fraudRiskPercentile = Math.round((overallRiskScore / confidenceScores.length) * 100). Here overallRiskScore is the SUM of all confidence scores and confidenceScores.length is the count. This computes the AVERAGE confidence and multiplies by 100. A person with ONE critical flag (0.85) scores 85. A person with FIVE high flags (each 0.85) also scores 85. This means a habitual multi-fraud actor is treated identically to a single-flag case — a fraud officer reading "85th percentile" would escalate them the same way. Risk must compound, not average.

FIX: Replace with independent-event risk combination: fraudRiskPercentile = Math.min(100, Math.round((1 - confidenceScores.reduce((p,c)=>p*(1-c),1)) * 100)). This gives: 1 flag at 0.85 → 85%; 5 flags each 0.85 → 99.9%.



// CURRENT (wrong — averages confidence):

fraudRiskPercentile = Math.round((overallRiskScore / confidenceScores.length) * 100);



// FIXED (compounds independent risk factors):

const combined = 1 - confidenceScores.reduce((acc, c) => acc * (1 - c), 1);

fraudRiskPercentile = Math.min(100, Math.round(combined * 100));



PART B — HIGH INCONSISTENCIES (Will Be Caught by Technical Reviewers)



Inconsistency B1 — Soft-Landing Formula in Assumptions vs. Code

HIGH

gen_assumptions.cjs documents a different formula than the code actually uses

gen_assumptions.cjs Soft-Landing Formula field says: max(300, (agi - 131000) x 0.01) — a 1% slope on income ABOVE the threshold. AlgorithmSimulation.js line 309 applies: Math.max(300, Math.round((agi * 0.0275) / 12)) — 2.75% of TOTAL AGI. At AGI KSh 150,000: assumption formula gives max(300, 190) = KSh 300; code gives max(300, 344) = KSh 344. A KIPPRA economist or CAJ technical reviewer who reads the assumptions register and then checks the code will find a KSh 44/month discrepancy at this income level. This destroys credibility.

FIX: Fix the gen_assumptions.cjs formula to match the code: "Statutory 2.75% flat rate applied to total AGI, annualised and divided by 12. At AGI 131,000, this produces exactly KSh 300.21/month — a mathematically smooth transition with no cliff." Delete the incorrect 1% formula.



Inconsistency B2 — 15% Compliance Uplift Is Fabricated

HIGH

revenueImpact calculation uses an undocumented 15% compliance bonus

AlgorithmSimulation.js line 346: revenueImpact: (monthlyContribution * 12) * 1.15 // Simulating 15% increased compliance due to fairness. This 1.15 multiplier appears nowhere in gen_assumptions.cjs. If presented to Cabinet Secretaries, KIPPRA economists, or SHA officials, this number will be immediately challenged: "Where does the 15% come from?" There is no ILO study, no Rwanda comparison, no actuarial basis cited.

FIX: Remove the multiplier entirely and report raw revenue impact. Add a SEPARATE sensitivity analysis table (which already exists in generateRevenueStressTest()) showing compliance at 30/45/60/75% — those ARE sourced to NHIF baseline, ILO, Rwanda CBHI. Let the table speak for itself.



Inconsistency B3 — KNBS Citation "2024" Does Not Exist

HIGH

gen_assumptions.cjs cites "KNBS Household Budget Survey 2024" which has not been published

The gen_assumptions.cjs cites "KNBS Household Budget Survey 2024" for the cost-of-living figures (KSh 18K, 10K, 4K). The KIHBS 2024/25 was launched as a pilot in April 2025 and is still in data collection as of June 2026. It has no published results. The most recent completed household survey is the Kenya Continuous Household Survey (KCHS) 2022. A KIPPRA economist or academic reviewer will instantly identify this as a non-existent citation — equivalent to citing a study that was never published.

FIX: Change citation to: "KCHS 2022 (KNBS Kenya Continuous Household Survey), consumer price index adjusted to 2025/26 using KNBS Monthly CPI." The KCHS 2022 shows food poverty lines of KSh 2,668/month rural and KSh 3,521/month urban — which, inflation-adjusted and including non-food costs, supports the KSh 4K rural and 10-18K urban CoL figures.



Inconsistency B4 — testEqualizedOdds Tests the Wrong Model

HIGH

The fairness test measures the current model's bias, not the proposed model's equalized odds

AlgorithmSimulation.js lines 1204-1207 in testEqualizedOdds: shouldBeIndigent = proposedResult.isIndigent === true; wasChargedAsNonIndigent = currentResult.isIndigent === false. This counts as a "false positive" any case where the proposed model says indigent AND the current model says not-indigent. This measures how often the CURRENT model overcharges people the proposed model would help — i.e., the current model's bias. It does NOT test whether the proposed model itself has disparate error rates across demographic groups (which is what "equalized odds" means in ML fairness literature). If you present this as "the proposed model passes equalized odds," a machine learning expert will refute it.

FIX: Rename the function to testCurrentModelDisparityByCounty() to accurately describe what it does. To genuinely test the proposed model's equalized odds, you need a ground-truth dataset (e.g., KCHS 2022 verified income data) and compare the proposed model's classification error across demographic groups against that truth. Document this as a requirement for the pilot study.



PART C — MEDIUM ISSUES (Fix Before Technical Submission)



Issue C1 — Transaction Count Assumption Not Documented

MEDIUM

The 80-transaction-per-month assumption for subsistence detection has no source

AlgorithmSimulation.js line 111: avgTransactionSize = d.grossMpesaMonthly / 80 — assumes 80 M-Pesa transactions per month when transactionCount is unavailable. This hidden assumption affects who qualifies for the Subsistence Digital User 60% balance discount. If someone actually makes 20 large transactions (KSh 1,250 each vs KSh 25,000/month), they would NOT qualify because 1,250 > 450 threshold. But the code assumes 80 transactions giving KSh 312 average, which DOES qualify. The 80-transaction assumption over-identifies subsistence users.

FIX: Add to gen_assumptions.cjs: "Default transaction count assumption: 80 per month (based on Safaricom M-Pesa transaction frequency data for informal merchants; source: Safaricom Annual Report FY2024 digital payments section)." Verify this against actual data before submission. Alternatively, collect transaction count as a field in the UI.



Issue C2 — Car Value Added to baseIncome Before CoL/Dependency Deductions

MEDIUM

Asset values flow through CoL and dependency deductions in an economically unintuitive way

Lines 173-191 add car and motorcycle values to baseIncome, then at line 223 agi = baseIncome. The CoL deduction (e.g., KSh 216,000/year for Nairobi), dependency deductions (8% per dependent), and CHE/PWD deductions all come off the combined income + asset value. This means a poor Nairobi renter with a cheap old car (KSh 210,000 depreciated = KSh 126,000 after 40% depreciation) could have their entire car value absorbed by the CoL deduction, making the car contribute zero to their AGI. While this may accidentally produce correct results, it is not explicitly documented and a careful auditor will question why CoL deductions are subtracted from asset values rather than income.

FIX: Add a code comment block explaining the design choice: the AGI model treats car/motorcycle value as a proxy for cumulative earned income saved into a capital asset, not as liquid income — therefore it flows into the same AGI pool that all living costs are deducted from. This is consistent with World Bank PMT asset annualization methodology. Document this in gen_assumptions.cjs under Asset Valuation Parameters.



Issue C3 — verify_logic.js Sanity Check Produces False Positives

MEDIUM

A logic test incorrectly flags valid cases as "Proposed charges MORE than Lasso for unverified user"

verify_logic.js check: if (proposed.monthlyContribution > lassoMonthly && !d.hasKraPin && !d.isNtsaVerified). This assumes the proposed model should always be cheaper for unverified users. But a Nairobi renter with moderate M-Pesa retained balance and a car (even an old one) could pay slightly more under AGI if the Lasso model under-counted their assets. This is intentional — the proposed model sometimes correctly charges MORE than the current broken model. The sanity check will generate false warnings that look like bugs during a demo.

FIX: Remove or refine this check. Replace with: if (proposed.monthlyContribution > lassoMonthly && proposed.isIndigent) issues.push("Indigent but charged above 300 — impossible state!"). Keep the indigent-but-not-300 check, which is genuinely impossible.



PART D — STRATEGIC GAPS (Strengthen Before Policy Submission)



Gap D1 — The KSh 11B Fraud is Conflated With PMT Reform Scope

INFO

The exec summary links the hospital fraud to PMT evasion — a SHA official will immediately correct this

The gen_exec_summary.cjs says the KSh 11B fraud "showed the system is bleeding money through undetected evasion." But the KSh 11B was from rogue hospitals fraudulently billing SHA on the claims side (fake surgeries, inflated procedures) — not from citizens evading enrollment contributions. SHA officials including CS Aden Duale and CEO Mercy Mwangangi are very clear on this distinction. If a SHA official is in the room and hears this conflated, they will push back hard and it will derail the rest of the presentation.

FIX: Reframe in all documents: "The KSh 11B hospital fraud and the PMT overcharging crisis are two sides of the same systemic failure. Adverse selection from overcharging drives a sicker enrolled pool, which incentivises hospital upcoding. The AGI model addresses enrollment-side integrity; it must be paired with separate claims-side fraud controls already underway."



Gap D2 — The IDinsight Pre-Deployment Report Is Your Strongest Evidence

INFO

The IDinsight warning ("inequitable, particularly for low-income households") is not prominently featured

The IDinsight report — obtained through access-to-information requests by Lighthouse Reports — is the most powerful evidence in your argument. It proves SHA knew the algorithm was flawed before launch and deployed it anyway. This is the difference between a policy mistake and a deliberate policy decision. The current documents mention "Error by Design" prominently but do not foreground the IDinsight report. In a CAJ filing context, this is the key document that shows institutional knowledge of harm.

FIX: Add a dedicated paragraph in the executive summary: "Before launch, international consultants IDinsight warned SHA that the PMT model was inequitable for low-income households and highly likely to miscalculate contributions without updated training data. This report was not made public. It was obtained through access-to-information requests. Its existence is why this reform is urgent, not merely desirable."



Gap D3 — No Section Addressing SHA's Own Reform Commitment

INFO

SHA has publicly committed to refining the MTI — aligning with this commitment makes the proposal collegial rather than adversarial

SHA CEO Mercy Mwangangi and CS Aden Duale have both made public statements committing to "continuously refining the means testing tool" (SHA press statement, May 6, 2026). The proposal documents treat SHA as an adversary to be overcome rather than an institution that has already signalled willingness to reform. In a CAJ or Parliamentary Health Committee context, framing this as adversarial is strategically weaker than framing it as "we are offering the technical specification SHA has already committed to seeking."

FIX: Add to executive summary opening: "SHA has publicly committed to refining its Means Testing Instrument. This proposal provides the technical specification for that refinement — a concrete, constitutionally-grounded, actuarially-sound replacement that can be piloted within 6 months without requiring legislative amendment."



Gap D4 — API MoU Requirement Is a Show-Stopper That Needs Acknowledged Up Front

INFO

The algorithm's fraud resistance completely depends on APIs that do not yet have legal frameworks

The fraud engine (18 flags) and the triangulation model both assume live access to KRA income data, NTSA TIMS vehicle registry, and Safaricom M-Pesa retained balance APIs. None of these APIs have formal MoUs with SHA. The Error by Design investigation does not mention these APIs existing. If asked "how does this work in practice today?", there is no good answer unless this is acknowledged upfront.

FIX: Create a dedicated implementation prerequisites section (already added to the bulletproof exec summary): Step 1 must always be the API MoU framework. Frame it positively: "Three MoUs. Six months. This is the only legislative/executive action required before the algorithm can be deployed at full accuracy."



Gap D5 — The Subsidy Fiscal Tradeoff Is Not Addressed

INFO

If the proposed model correctly reclassifies more people as indigent, the subsidy cost increases proportionally

The revenue stress test hardcodes a KSh 3.2B/month indigent subsidy cost. But the entire purpose of the AGI model is to correctly classify more previously-overcharged people as indigent. If the model correctly reclassifies 30% more households as indigent (which is likely given the Lighthouse Reports audit finding >50% exclusion errors in the bottom quartile), the government subsidy obligation increases by 30% too. This fiscal tradeoff is never addressed in any of the generated documents. A Treasury official reviewing the proposal will ask: "If you fix the classification errors, how much MORE does the indigent subsidy programme cost?"

FIX: Add a fiscal tradeoff section: "Correctly classifying more households as indigent increases the subsidy obligation. At the current KSh 300/month subsidy rate, reclassifying 1M additional households costs KSh 300M/month. However: (a) the SHIF collected KSh 57.7B against KSh 96.1B spending — the current system is already unsustainable; (b) higher compliance from fairly-assessed non-indigent contributors more than offsets the increased subsidy at 45%+ compliance; (c) the alternative — continued adverse selection — produces a MUCH larger fiscal hole." This is the actuarial argument for reform.
